### How signing protects against tag tampering

Docker tags are mutable - anyone with push access can redirect a tag to a completely different image. 
Cosign signs the **content-addressable digest** (SHA-256 of the manifest), not the tag. 
When an attacker replaces the image under a tag, the new digest has no associated signature, 
so `cosign verify` immediately fails with "no signatures found". The original signed digest remains verifiable 
regardless of what the tag points to. This is why signing and verifying by digest (not by tag) is the correct practice.

**Subject digest** is the SHA-256 hash of the image manifest included in the signature envelope as the subject. 
It uniquely and immutably identifies the exact bytes of the image - any change to the image 
content produces a different digest, automatically invalidating the signature.

### How attestations differ from signatures

A **signature** answers "is this image authentic?" - it is a cryptographic proof that a specific key
signed a specific digest. It contains no semantic information about the image content.

An **attestation** answers "what claims do we make about this image?" - it wraps a structured predicate 
(SBOM, provenance, vulnerability scan results) in an in-toto Statement envelope and signs that envelope. 
The predicate carries semantic metadata that can be audited and verified independently.

### What the SBOM attestation contains

The CycloneDX SBOM attestation (generated by Syft v1.42.1) contains the full bill of materials for the 
juice-shop container: **3533 software components** including Node.js packages (from `package.json`), 
OS-level packages (Alpine apk), their versions, declared licenses, CPEs, and PURLs. This enables downstream
vulnerability scanning (e.g. Grype) and license compliance checks without running the container.

### What provenance attestations provide for supply chain security

A provenance attestation records the circumstances of how an artifact was produced: the builder identity, 
build type, invocation parameters, and build timestamp. For supply chain security this provides:
- An auditable chain from source code to final artifact
- A basis for SLSA (Supply-chain Levels for Software Artifacts) compliance checks
- Evidence that the build was not tampered with after it left the CI system
- Traceability when investigating a security incident - you can determine exactly what was used to produce a given image digest

### Use cases for signing non-container artifacts

- **Release binaries** - sign compiled executables so users can verify they downloaded the authentic build before running it
- **Configuration files** - sign Kubernetes manifests, Helm charts, or Terraform modules to prevent unauthorized modification in GitOps pipelines
- **Language packages** - sign `.whl`, `.jar`, or `.gem` archives before publishing to package registries
- **Checksums files** - sign a `sha256sums.txt` for a release, providing a compact and verifiable manifest over many files
- **ML models and datasets** - sign to prove provenance and enable reproducibility audits

### How blob signing differs from container image signing

Container image signing stores the signature as an OCI artifact in the same registry as the image.
The signature is automatically discoverable - `cosign verify image:tag` queries the registry for a matching 
signature artifact. No extra files need to be distributed alongside the image.

Blob signing produces a standalone **bundle file** (JSON containing the signature, digest, and timestamp) that 
must be distributed manually alongside the artifact. The consumer must explicitly provide `--bundle` on verification. 
There is no registry to act as the signature store - the bundle is the only proof of signing.
